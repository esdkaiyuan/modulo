# Unified Mono and Color Modulo Design

## Goal

Make every modulo workspace support two consistent processing modes:

- Mono: 1-bit bitmap output.
- Color: RGB565, RGB888, or 16-color indexed output.

Every encoding must export C arrays, HEX text, and BIN files. Image, batch,
animation, video, font, and handdraw pages must use the same encoding rules.

## Scope

This design covers the shared processing and export engines, the six tool
stores, their page controls and previews, and automated regression coverage.
It preserves the existing page layouts, media import paths, drawing history,
and hash-based navigation.

## Shared Types

The shared engine exposes these concepts:

```ts
export type ModuloMode = 'mono' | 'rgb565' | 'rgb888' | 'palette16';
export type ExportFormat = 'c-array' | 'hex' | 'bin';
export type Rgb565ByteOrder = 'msb-first' | 'lsb-first';
export type Rgb888Order = 'rgb' | 'bgr';

export interface ColorEncodeResult {
  pixelBytes: Uint8Array;
  paletteBytes?: Uint8Array;
  previewImageData: ImageData;
  bitsPerPixel: 16 | 24 | 4;
}
```

Mono mode continues using the existing scan direction, bit order, polarity,
threshold, inversion, and dithering types. Color mode uses scan direction but
does not expose mono-only bit order or polarity controls.

## Architecture

### Image processing

`imageProcessor` owns crop, resize, alpha compositing, brightness, contrast,
and grayscale conversion. Both mono and color paths operate on the same
cropped and resized target image. The source media remains immutable.

Transparent pixels are composited against the selected background color before
RGB565, RGB888, or palette quantization. A transparent font or drawing
background uses the configured transparent-pixel fallback during binary
encoding because these three output formats do not carry alpha.

### Mono encoding

`bitmapEncoder` remains the single 1-bit encoder. It supports horizontal and
vertical traversal in both directions, MSB-first and LSB-first packing,
positive and negative polarity, and dimensions that are not multiples of
eight.

### Color encoding

A shared `colorEncoder` accepts target-sized `ImageData` and traversal options.
It produces encoded bytes plus an encoding-roundtrip preview.

- RGB565 stores two bytes per pixel. `msb-first` writes the high byte first;
  `lsb-first` writes the low byte first.
- RGB888 stores three bytes per pixel. `rgb` writes R, G, B; `bgr` writes B,
  G, R.
- Palette16 generates 16 RGB565 palette entries and packs two 4-bit pixel
  indices per byte. The first traversed pixel occupies the high nibble. An odd
  final pixel uses a zero low-nibble pad.

Palette colors are derived from the processed source instead of using a fixed
hard-coded palette. Animation and video use one palette derived from all
selected frames so colors remain stable during playback.

### Output formatting

`outputFormatter` accepts encoded results and produces:

- C arrays with resolution, encoding, byte order, frame count, and timing
  metadata where applicable.
- HEX text containing the exact bytes in binary export order.
- BIN blobs containing the exact encoded byte stream.

Palette16 C and binary output place the 32-byte RGB565 palette before packed
indices. Multi-frame binary output places shared metadata in the C/HEX
representation and uses a documented deterministic byte layout for the BIN
payload. File extensions are `.h`, `.hex.txt`, and `.bin`.

## Page Behavior

All six pages expose a segmented Mono/Color selector. Color mode exposes the
color encoding selector and only the controls relevant to that encoding.
Mono-only threshold, dithering, inversion, bit order, and polarity controls are
hidden in color mode. Scan direction remains available in both modes.

All previews display decoded encoded output rather than the untouched source,
so quantization, threshold, channel order, and palette effects are visible.
Changing an encoding option reprocesses the current source without discarding
uploads, decoded frames, or handdraw history.

### Image

The selected crop is extracted and resized before either mono or color
encoding. Generated byte counts and previews must match the requested target
dimensions. RGB565 and RGB888 support their byte/channel order controls.

### Batch

One settings set applies to the queue. Each item stores its own encoded result,
preview, source text, error, and progress. C and HEX results can be merged in
queue order. BIN results are downloaded per item to avoid an ambiguous merged
container. One failed item does not stop other items and remains retryable.

### Animation

Selected GIF frames are resized and encoded in source order. Color palette mode
uses one shared palette for every selected frame. C output retains frame delays;
HEX output documents the same ordering; BIN output is deterministic and uses
the shared palette once followed by frame data.

### Video

Sampled video frames follow the animation encoding rules. Output retains the
configured FPS and frame ordering. Palette mode uses one palette for all
processed frames.

### Font

Color mode adds foreground color, background color, and transparent-background
controls. Text is rendered to RGBA, and glyph coverage blends foreground and
background before encoding. The existing threshold and inversion path remains
for mono mode.

### Handdraw

Color mode converts stored pixel colors directly to RGBA. Empty cells use the
configured transparent-pixel background during encoding. Mono mode treats
occupied cells as set pixels and preserves the existing drawing, fill,
symmetry, undo, and redo behavior.

## Validation and Errors

Processing validates positive integer target dimensions, source availability,
frame ranges, and output-size estimates. Empty input produces an empty result
and disables download. Invalid dimensions and excessive output estimates show
a clear page-level error instead of attempting allocation.

Batch processing isolates errors per file. Invalid GIF/video frame collections
produce no downloadable result. An export cannot proceed until the current
settings have a successfully encoded result.

## Testing

Engine tests cover:

- Mono traversal, polarity, bit order, and partial final bytes.
- RGB565 known colors, byte order, traversal, and alpha compositing.
- RGB888 RGB/BGR order, traversal, and byte length.
- Palette generation, closest-color indexing, shared multi-frame palettes,
  nibble packing, and odd-pixel padding.
- C, HEX, and BIN byte equivalence for all encodings.

Store tests cover image crop/resize correctness, batch failure isolation,
animation delays, video FPS, shared palettes, font color blending, handdraw
colors and transparent cells, filename extensions, and mode changes that retain
source state.

Page integration tests require every workspace to expose both processing modes,
the three color encodings, the three export formats, and encoding-specific
controls without rendering irrelevant options.

## Acceptance Criteria

- Every tool page can generate valid mono output.
- Every tool page can generate RGB565, RGB888, and Palette16 output.
- Every encoding exports C, HEX, and BIN with identical underlying bytes.
- RGB565 byte order and RGB888 channel order match the selected setting.
- Palette16 indices are packed at 4 bits per pixel and multi-frame palettes are
  stable across frames.
- Encoded previews match exported content and target dimensions.
- Existing media import, frame selection, drawing, and navigation behaviors do
  not regress.
- The full Vitest suite and production build pass.
